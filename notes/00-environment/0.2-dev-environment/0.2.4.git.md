# 0.2.4 了解 Git 基础：clone、branch、commit、merge、rebase、PR 工作流

> [← 返回 0.2 目录](./README.md) · [知识库地图](../../../knowledge-map/01-foundation-stages-0-2.md#02-开发环境)

## 一句话理解

**Git** 是分布式版本控制系统：记录代码每一次变更，支持 **分支并行开发** 与 **协作合并**；日常流程是 **clone → branch → commit → push → Pull Request → merge**。

---

## 为什么 Web 开发必须会 Git

| 场景 | Git 的作用 |
| --- | --- |
| 个人练习 | 随时回退、对比历史 |
| 团队协作 | 多人改同一仓库不互相覆盖 |
| Code Review | PR 上讨论后再合入 main |
| CI/CD | 推送触发自动测试与部署 |
| 开源贡献 | Fork → 改 → PR |

本仓库 **WebDev_Compass** 本身即 Git 管理；Cursor/VS Code 内置 **源代码管理** 面板可视化操作。

---

## 核心概念

```
工作区 (Working Directory)  →  暂存区 (Staging / Index)  →  本地仓库 (.git)
                                                              ↕
                                                         远程仓库 (origin)
```

| 概念 | 说明 |
| --- | --- |
| **仓库（Repository）** | 含完整历史与 `.git` 目录的项目 |
| **commit** | 一次快照，含作者、时间、说明 |
| **branch** | 指向某 commit 的可移动指针 |
| **remote** | 远程地址，通常叫 `origin` |
| **HEAD** | 当前所在分支/commit |

## 初次配置

```bash
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
git config --global init.defaultBranch main
git config --global pull.rebase false   # 或 true，团队统一即可
```

查看配置：`git config --list --global`

---

## clone：获取远程仓库

```bash
git clone https://github.com/org/repo.git
cd repo
```

等价于：下载项目 + 自动 `git remote add origin` + 检出默认分支。

SSH 方式（需配置公钥）：

```bash
git clone git@github.com:org/repo.git
```

## 日常三板斧：status / add / commit

```bash
git status              # 哪些文件改了、是否已暂存
git add index.html      # 暂存单个文件
git add .               # 暂存所有变更（注意 .gitignore）
git commit -m "feat: add login form"
```

### 写好 commit message

推荐 **Conventional Commits** 前缀：

| 前缀 | 含义 |
| --- | --- |
| `feat:` | 新功能 |
| `fix:` | 修复 bug |
| `docs:` | 文档 |
| `style:` | 格式（不影响逻辑） |
| `refactor:` | 重构 |
| `test:` | 测试 |
| `chore:` | 构建/工具变动 |

示例：`fix: handle empty response in user list API`

## branch：并行开发

```bash
git branch                  # 列出本地分支
git branch feature/login    # 创建分支
git checkout feature/login  # 切换分支
git checkout -b feature/login   # 创建并切换（常用）
git switch -c feature/login     # 新语法，同上
```

### 命名建议

- `feature/xxx`、`fix/xxx`、`docs/xxx`
- 小写、短横线分隔，见名知意

### 查看分支关系

```bash
git log --oneline --graph --all
```

---

## 与远程同步

```bash
git fetch origin              # 拉取远程更新，不合并
git pull origin main          # fetch + merge（或 rebase）
git push -u origin feature/login   # 首次推送并设置 upstream
git push                      # 之后直接 push
```

`-u`（`--set-upstream`）建立本地分支与远程跟踪关系。

---

## merge：合并分支

在 `main` 上合入 `feature/login`：

```bash
git checkout main
git pull origin main
git merge feature/login
```

若 **无冲突**，Git 自动产生 **merge commit**（fast-forward 或三方合并）。

### 解决冲突

冲突文件含标记：

```
<<<<<<< HEAD
main 上的内容
=======
feature 上的内容
>>>>>>> feature/login
```

手动编辑保留正确内容 → `git add` → `git commit`。

Cursor/VS Code 提供 **Merge Editor** 可视化解决。

---

## rebase：变基（整理历史）

```bash
git checkout feature/login
git rebase main
```

把 feature 上的 commit **挪到** main 最新 commit 之后，历史呈 **直线**，更整洁。

### merge vs rebase

| | merge | rebase |
| --- | --- | --- |
| 历史 | 保留分叉与 merge commit | 线性 |
| 安全性 | 不改已有 commit | 改写 commit hash |
| 适用 | 公共分支、发布分支 | **本地** feature 整理后 push |

**黄金法则**：不要对 **已 push 且他人基于其开发** 的分支做 rebase。

### pull 时的 rebase

```bash
git pull --rebase origin main
```

先拉远程 main，再把当前分支 commit 接在后面，减少无意义 merge commit。

## Pull Request（PR）工作流

GitHub / GitLab / Gitee 上的标准协作：

```
1. 从 main 拉最新代码
2. 创建 feature 分支
3. 本地 commit 并 push 到 origin
4. 在平台上开 Pull Request（base: main ← compare: feature）
5. CI 跑测试、同事 Review
6. 修改意见 → 继续 commit push（PR 自动更新）
7. Squash merge / Merge commit / Rebase merge（团队策略）
8. 删除已合并的远程分支
```

### 你在 PR 里应提供

- 清晰的 **标题与说明**（改了什么、为什么）
- **Test plan**：如何验证
- 关联 Issue 编号（若有）

## .gitignore

不应提交的文件：

```
node_modules/
dist/
.env
.DS_Store
*.log
```

Web 项目几乎必忽略 `node_modules/` 与构建产物 `dist/`。

## 常用补救命令

| 场景 | 命令 |
| --- | --- |
| 撤销工作区修改 | `git checkout -- file` 或 `git restore file` |
| 取消暂存 | `git reset HEAD file` 或 `git restore --staged file` |
| 修改最后一次 commit 说明 | `git commit --amend`（未 push 时） |
| 暂存当前工作 | `git stash` / `git stash pop` |
| 查看某行谁改的 | `git blame file.js` |

## 📱 移动端对照

| Git / GitHub | 移动端 |
| --- | --- |
| Git 版本控制 | 同样用 Git（Android/iOS 项目都在 Git 里） |
| `package.json` + lock | Gradle lockfile、Podfile.lock |
| PR Code Review | 完全相同流程 |
| CI on push | GitHub Actions / GitLab CI 同样触发移动端构建 |

移动端额外还有 **签名证书、Flavor、应用商店发布**，但 **分支与 PR 协作模型一致**。

## 常见误区

| 误区 | 正确理解 |
| --- | --- |
| 「commit 越多越好」 | 应 **原子化**：一个 commit 做一件事 |
| 「main 上直接改最快」 | 团队项目应走分支 + PR |
| 「rebase 总是比 merge 好」 | 仅适合整理 **本地未共享** 历史 |
| 「push 了就不能改」 | 可 revert 或新 commit 修复；避免 force push 公共分支 |
| 「.gitignore 能忽略已跟踪文件」 | 需先 `git rm --cached` 再 ignore |
