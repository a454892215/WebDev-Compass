# 0.2.5 📱 移动端对照：npm ≈ pub / CocoaPods / Gradle；Vite dev server ≈ Hot Reload

> [← 返回 0.2 目录](./README.md) · [知识库地图](../../../knowledge-map/01-foundation-stages-0-2.md#02-开发环境)

## 一句话理解

Web 开发环境里的 **包管理器** 和 **开发服务器热更新**，与 Android / Flutter / React Native 中的 **依赖管理** 与 **Hot Reload / Hot Restart** 是同一类问题在不同平台上的实现——搞清对照表，能大幅缩短上手时间。

---

## 总览对照表

| Web（本节 0.2） | Android | Flutter | React Native |
| --- | --- | --- | --- |
| Node.js | JDK + Android SDK | Dart SDK | Node.js + JDK (Android) |
| npm / pnpm | Gradle (Maven) | pub | npm / yarn |
| `package.json` | `build.gradle` / `libs.versions.toml` | `pubspec.yaml` | `package.json` |
| `node_modules/` | Gradle cache | `.pub-cache` | `node_modules/` |
| VS Code / Cursor | Android Studio | VS Code / Android Studio | VS Code |
| Chrome DevTools | Layout Inspector + Logcat | DevTools (Flutter) | Flipper / Dev Menu |
| Git + PR | 相同 | 相同 | 相同 |
| Vite dev server + HMR | Apply Changes / Live Edit | Hot Reload | Fast Refresh |

## 依赖管理：npm ≈ pub / CocoaPods / Gradle

### 共同模式

```
声明依赖清单 → 工具解析版本 → 下载到本地缓存 → 构建时链接进项目
```

### Web：npm / pnpm

```json
// package.json
{
  "dependencies": {
    "axios": "^1.7.0"
  }
}
```

```bash
npm install
```

-  registry：npmjs.com
- 锁文件：`package-lock.json` / `pnpm-lock.yaml`

### Flutter：pub

```yaml
# pubspec.yaml
dependencies:
  dio: ^5.4.0
```

```bash
flutter pub get
```

-  registry：pub.dev
- 锁文件：`pubspec.lock`

### Android：Gradle

```kotlin
// build.gradle.kts
dependencies {
    implementation("com.squareup.retrofit2:retrofit:2.9.0")
}
```

```bash
./gradlew build
```

-  仓库：Maven Central、Google Maven
-  版本目录：`libs.versions.toml`（现代项目）

### iOS：CocoaPods / SPM

```ruby
# Podfile
pod 'Alamofire', '~> 5.8'
```

```bash
pod install
```

### 关键差异

| 维度 | Web | 移动端 |
| --- | --- | --- |
| 依赖产物 | JS 模块，运行时由打包器打进 bundle | AAR、framework、native .so |
| 原生代码 | 一般不直接管 | NDK、Pod 常含 C/C++/Swift |
| 安装体积 | `node_modules` 可能很大但只在开发机 | 影响 App 体积需裁剪 |

**迁移心法**：把 `npm install` 当成 `flutter pub get` / `./gradlew build` 之前的 **必做步骤**；把 lock 文件当成 **团队必须提交** 的版本真相。

## 编辑器：VS Code / Cursor ≈ Android Studio / Xcode

| 能力 | Web | 移动端 |
| --- | --- | --- |
| 智能提示 | TS/JS LSP | Kotlin/Java/Swift/Dart LSP |
| 重构 | F2 重命名符号 | 同等 Refactor |
| 内置终端 | 跑 npm scripts | 跑 gradle、flutter |
| 调试 | Chrome 挂 Sources | 挂进程断点 |
| 插件 | ESLint、Prettier | ktlint、SwiftLint |

有 **Android Studio 经验** 的开发者：把 **Problems 面板** 当 Inspection Results，把 **Live Server** 当 **Run App 到模拟器** 的简化版（仅静态页）。

**Cursor 特有**：AI 辅助写 CSS/配置，类似 Android Studio 的 AI 实验功能，但 Web 侧往往更常用。

## 调试：Chrome DevTools ≈ Logcat / Flutter DevTools

| DevTools 面板 | 移动端近似 |
| --- | --- |
| **Console** | Logcat / `debugPrint` |
| **Network** | OkHttp Logging / Charles 抓包 |
| **Elements** | Layout Inspector / Widget 树 |
| **Application** | SharedPreferences / UserDefaults |
| **Sources 断点** | IDE Debugger |

Web **优势**：浏览器 + DevTools **零配置** 看 Network 与 DOM。

Web **注意**：没有「单一 Activity 栈」；路由是 URL + History API，见 [0.1.3 SPA vs MPA](../0.1-what-is-web-app/0.1.3.spa-vs-mpa.md)。

## 热更新：Vite HMR ≈ Hot Reload / Fast Refresh

### 开发环境 vs 生产环境（重要）

| 环境 | Web | 移动端 |
| --- | --- | --- |
| **开发** | Vite **HMR**：改 CSS/JS 后 **浏览器局部更新**，不整页刷新 | Flutter Hot Reload、RN Fast Refresh |
| **生产** | 用户访问的是 **构建后的静态文件**，更新 = **重新部署** | App Store / OTA（若启用 CodePush 等） |

**0.2.5 里的「热更新」指开发期 HMR**，不是让用户无感升级线上版本。

### Vite dev server 做了什么

```
源代码文件变更
    → Vite 编译受影响的模块
    → WebSocket 通知浏览器
    → 浏览器替换 ES 模块 / 注入新 CSS
    → 页面状态尽量保留（HMR API）
```

启动：

```bash
npm run dev   # 通常即 vite，默认 http://localhost:5173
```

### Flutter Hot Reload

改 Dart 代码 → `r` 或保存 → **Widget 树增量重建**，状态保留。

### React Native Fast Refresh

改 JS/TSX → Metro bundler 推送 → 组件热替换，类似 Web HMR。

### Android「热」相关（易混淆）

| 名称 | 是什么 |
| --- | --- |
| **Apply Changes** | 某些资源/方法改动快速部署到模拟器 |
| **Live Edit (Compose)** | Compose UI 实时预览 |
| **非** Google Play 热更新 | 替换 APK 需重新安装或应用内更新 |

与 Vite HMR **最接近** 的是 **RN Fast Refresh** 和 **Flutter Hot Reload**——都是 **开发循环** 加速。

## Git 与 PR：完全一致

无论 Web 还是移动端，现代团队都使用：

- `main` / `develop` 分支策略
- feature branch + Pull Request
- CI 在 push 时跑 lint / test / build

你在 Android 项目里用的 Git 流程 **原样适用** 于 Web 项目。

## 从移动端转 Web 的推荐路径

若你已有移动端经验，按此顺序对照学习 0.2：

1. **0.2.1**：把 Node/npm 当成 pub + Gradle 的合体工具链
2. **0.2.2**：继续用 VS Code，加 Web 插件即可
3. **0.2.3**：DevTools ≈ Logcat + Layout Inspector + Charles
4. **0.2.4**：跳过或速览（已会则直接自测）
5. 进入 **0.3 网络**：Retrofit/Dio 经验直接映射到 fetch/axios

## 常见误区

| 误区 | 正确理解 |
| --- | --- |
| 「Web 也能像 App 一样静默热更新用户」 | 生产 Web 需部署；开发 HMR ≠ 线上 OTA |
| 「npm 只像 CocoaPods」 | 还像 Gradle **脚本任务**（`npm run build`） |
| 「会用 Android Studio 就不学 DevTools」 | DOM/CSS/Network 是 Web 独有调试面 |
| 「node_modules 可以删了随时再下」 | 可以，但安装耗时；且 lock 文件保证版本一致 |
| 「Vite 只能用于 Vue」 | Vite 支持 React/Svelte/Vanilla 等 |
