# 0.3.7 RESTful API 设计风格与常见约定

> [← 返回 0.3 目录](./README.md) · [知识库地图](../../../knowledge-map/01-foundation-stages-0-2.md#03-网络基础web-开发者必备)

## 一句话理解

**REST** 是一种以 **资源** 为中心、用 **HTTP 方法表达操作** 的 API 设计风格。实际项目多为「遵循常见约定的 HTTP JSON API」，很少追求学术上的完全 REST。

---

## 核心思想

1. **资源** 用 URL 名词表示：`/users`、`/users/42`、`/users/42/orders`
2. **操作** 用 HTTP 方法：`GET` 读、`POST` 建、`PUT`/`PATCH` 改、`DELETE` 删
3. **无状态**：每个请求自带鉴权与必要上下文，服务器不依赖隐藏的「上一步对话」（Session 例外见工程实践）
4. **统一接口**：同类资源的路径与状态码语义一致，便于客户端推断

非 REST 对照：SOAP、纯 RPC（`/getUser`、`/deleteUser` 动词堆在路径里）——也能用，但 REST 约定更利于 HTTP 缓存与工具生态。

---

## URL 与方法约定

| 操作 | 方法 | 路径 | 成功状态码（常见） |
| --- | --- | --- | --- |
| 列表 | GET | `/api/users` | 200 |
| 详情 | GET | `/api/users/:id` | 200 |
| 创建 | POST | `/api/users` | 201 |
| 全量更新 | PUT | `/api/users/:id` | 200 |
| 部分更新 | PATCH | `/api/users/:id` | 200 |
| 删除 | DELETE | `/api/users/:id` | 204 或 200 |

嵌套资源：

```
GET /api/users/42/orders
POST /api/users/42/orders
```

过滤与分页放 **查询参数**：

```
GET /api/users?role=admin&page=2&pageSize=20&sort=-createdAt
```

原则：

- 路径用 **名词复数**；避免 `/getUsers`
- 用 **连字符或小写** 保持风格统一（团队二选一）
- 版本可选：`/api/v1/users` 或头 `Accept: application/vnd.myapi.v1+json`

## 请求与响应体

### 请求

```http
POST /api/users HTTP/1.1
Content-Type: application/json
Authorization: Bearer <token>

{"name":"Ada","email":"ada@example.com"}
```

### 成功响应

```json
{
  "id": 42,
  "name": "Ada",
  "email": "ada@example.com",
  "createdAt": "2026-07-15T10:00:00Z"
}
```

### 列表响应（常见包一层）

```json
{
  "data": [ ... ],
  "meta": { "page": 1, "pageSize": 20, "total": 100 }
}
```

### 错误响应（约定一例）

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "email is invalid",
    "details": [{ "field": "email", "issue": "format" }]
  }
}
```

配上合适状态码：`400` / `422` / `401` / `403` / `404` / `409` / `429`。

## 鉴权常见方式

| 方式 | 说明 |
| --- | --- |
| **Bearer Token** | `Authorization: Bearer ...`（JWT 或不透明 token） |
| **Cookie Session** | 浏览器自动带 Cookie；注意 CSRF |
| **API Key** | 服务间或简单脚本；勿塞进前端公开仓库 |

## 前端对接 checklist

```javascript
async function createUser(body) {
  const res = await fetch('/api/users', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      Authorization: `Bearer ${getToken()}`,
    },
    body: JSON.stringify(body),
  });
  const data = await res.json().catch(() => ({}));
  if (!res.ok) {
    throw Object.assign(new Error(data?.error?.message || res.statusText), {
      status: res.status,
      data,
    });
  }
  return data;
}
```

关注：路径是否符合约定、是否检查 `res.ok`、错误结构是否统一、幂等（重试 POST 可能双写）。

## REST 不是什么

| 说法 | 澄清 |
| --- | --- |
| 「有 JSON 就是 REST」 | REST 强调资源与方法语义，不取决于 JSON |
| 「必须四级 Richardson 成熟度」 | 工程上够用、一致即可 |
| 「不能有 RPC 风格端点」 | 复杂动作可用 `/orders/42:cancel` 或 `POST /orders/42/cancellation`（团队约定） |

GraphQL / gRPC 是其它风格：按场景选型，不互相取代。

## 📱 移动端对照

| Web fetch | Retrofit | Dio |
| --- | --- | --- |
| `GET /users` | `@GET("users")` | `dio.get('/users')` |
| `POST` + JSON body | `@POST` + `@Body` | `dio.post(..., data:)` |
| 查询参数 | `@Query` | `queryParameters` |
| 统一错误处理 | Interceptor / `HttpException` | Interceptor / `DioException` |

同一套 REST 后端可同时服务 Web 与 App。

## 常见误区

| 误区 | 正确理解 |
| --- | --- |
| 「用 POST 做所有事更简单」 | 丢掉缓存、语义与中间件约定的红利 |
| 「DELETE 不能有 body」 | 规范未严格禁止，但常用查询或路径传 id |
| 「201 和 200 无所谓」 | 约定清晰可减少前后端沟通成本 |
