## 总览对照表

| Web（本节） | Android | Flutter | React Native |
| --- | --- | --- | --- |
| URL / `URLSearchParams` | `HttpUrl` / `Uri` | `Uri` | `URL` / 自行拼 |
| `fetch` / axios | OkHttp / Retrofit | dio / http | fetch / axios |
| Cookie + Session | CookieJar + 服务端会话 | 同左 | 同左或 Token |
| localStorage | SharedPreferences | shared_preferences | AsyncStorage |
| sessionStorage | 内存 / Activity 作用域 | 内存 | 内存 |
| HTTPS / 证书 | 系统信任库 + 可选 Pinning | 同左 | 同左 |
| DNS / CDN | 同样依赖 | 同样依赖 | 同样依赖 |
| REST JSON API | `@GET`/`@POST` | `dio.get/post` | 同 Web |
| WebSocket | OkHttp WebSocket | web_socket_channel | ws 库 / 原生 |
| CORS | **无** | **无** | **无**（纯原生）；WebView/H5 **有** |

## HTTP 客户端：Retrofit / OkHttp / Dio ≈ fetch / axios

### 共同模式

```
拼 URL → 设 Method / Header / Body → 发送 → 解析状态码与 JSON → 错误分支
```

### 并列示例：GET 列表

**Web（fetch）**

```javascript
const res = await fetch('https://api.example.com/users?page=1', {
  headers: { Authorization: `Bearer ${token}` },
});
if (!res.ok) throw new Error(String(res.status));
const data = await res.json();
```

**Android（Retrofit 示意）**

```kotlin
interface Api {
  @GET("users")
  suspend fun users(@Query("page") page: Int): List<User>
}
```

**Flutter（dio）**

```dart
final res = await dio.get('/users', queryParameters: {'page': 1});
final data = res.data;
```

### axios vs Retrofit

| 能力 | axios | Retrofit |
| --- | --- | --- |
| 拦截器 | `interceptors` | OkHttp Interceptor |
| 基地址 | `baseURL` | `@BaseUrl` / Retrofit Builder |
| 取消 | `AbortController` / CancelToken | `Call.cancel` / Coroutine Job |
| 类型 | TS 接口 | 数据类 + Converter |

有 Retrofit 经验：学 axios/fetch 主要是 API 形状不同，**状态码与 REST 约定完全可复用**。

## 存储：SharedPreferences ≈ localStorage

| 能力 | Web localStorage | SharedPreferences / shared_preferences / AsyncStorage |
| --- | --- | --- |
| 键值 | 字符串 | 多为原始类型 / 字符串 |
| 容量 | ~5MB 量级 | 适合小配置，大数据另寻 |
| 敏感数据 | 易被 XSS 读 | 需加密存储或 Keystore |
| 与请求关系 | 不自动上传 | 不自动上传 |

Cookie 会话在 App 里常变成：**Token 存安全存储 + 拦截器写 Authorization**，而不是依赖浏览器 Cookie 自动携带（除非 CookieJar 专项配置）。

---

## 无 CORS：最大心智差异

| | 浏览器 Web | 原生 App |
| --- | --- | --- |
| 跨域读响应 | 默认拦截，需 CORS | 默认可请求任意 URL |
| 安全边界 | 同源策略 + CORS + Cookie 策略 | App 签名、权限、证书、自有鉴权 |

从 Android 转 Web 时，**最容易踩坑的是 CORS 与 Cookie 的 SameSite**——协议懂了仍会被浏览器拦。回顾 [0.1.4](../0.1-what-is-web-app/0.1.4.same-origin-policy.md)。

WebView 内嵌 H5：**重新变成浏览器规则**。

---

## HTTPS / 证书钉扎

- Web：用户看得到证书错误页；前端很少做 Pinning
- App：校检失败直接异常；金融类常见 **Certificate Pinning**
- 调试抓包需在系统中安装用户证书或关 Pinning——与 Charles 调试 Web 类似但配置点不同

## 实时：WebSocket 与推送

| 场景 | Web | App |
| --- | --- | --- |
| 前台实时 | WebSocket / SSE | 同协议 |
| 进程在后台 | 连接易被挂起 | 改用 FCM / APNs 系统推送 |
| 重连 | 业务自建 | 同左 |

## 迁移建议（给移动端同学）

1. **先搬 REST 与错误码契约**——后端往往已是同一套。
2. **用 DevTools Network 代替 Charles 看浏览器独有问题**（CORS、预检、SameSite）。
3. **登录态**：弄清项目是 Cookie 会话还是 Bearer Token，再对照 App 里的 Token 存储。
4. **Storage**：localStorage ≈ SharedPreferences，但 **不要把 Web 当安全保险柜**。
5. **协议升级（h2/h3）、DNS、CDN、TLS**：客户端无感或配置在基建侧，概念两端通用。
