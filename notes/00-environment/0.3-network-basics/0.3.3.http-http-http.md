# 0.3.3 HTTP/1.1、HTTP/2、HTTP/3 基本概念与差异

> [← 返回 0.3 目录](./README.md) · [知识库地图](../../../knowledge-map/01-foundation-stages-0-2.md#03-网络基础web-开发者必备)

## 一句话理解

三个版本的 **语义层（方法、状态码、头）大体相同**，差别主要在 **如何在连接上传递多个请求**：HTTP/1.1 易阻塞，HTTP/2 用多路复用，HTTP/3 再换成基于 UDP 的 QUIC。

---

## 为什么需要演进

加载一个页面往往几十上百个请求（HTML、CSS、JS、图片、字体、API）。瓶颈常在：

1. 连接建立慢（TCP + TLS 握手）
2. 同一连接上请求互相堵（队头阻塞）
3. 丢包时整条连接受影响

版本升级主要解决 **传输效率与丢包韧性**，不是改业务 API 写法。

---

## HTTP/1.1（至今仍普遍）

特点：

- 文本协议、易读
- **Keep-Alive**：同一 TCP 连接可复用多次请求（默认开启）
- **管线化（pipelining）** 实践中基本弃用

痛点：**队头阻塞（Head-of-Line Blocking）**——同一连接上前一个响应没回来，后面请求也卡住。浏览器靠 **每域名 6 左右并行 TCP 连接** 缓解。

工程上曾用：域名分片（sharding）、雪碧图、内联小资源——很多在 HTTP/2 下不再必要。

---

## HTTP/2

特点：

1. **二进制分帧**：消息切成帧传输
2. **多路复用**：单连接上并行多个请求/响应流（Stream）
3. **头部压缩（HPACK）**：重复头（如 Cookie、User-Agent）开销小
4. **服务端推送（Server Push）**：实际采用少，可了解即可
5. 生产几乎总是 **加密（h2 over TLS）**

队头阻塞改善：应用层不再因「前一个慢响应」堵住后续流；但 **TCP 丢包仍可能拖慢整条连接**。

前端几乎无感知：仍用 `fetch`；协商由浏览器与服务器 ALPN 完成。

---

## HTTP/3

特点：

1. 传输层改用 **QUIC（基于 UDP）**，而非 TCP
2. 连接迁移更好（弱网、切 Wi‑Fi ↔ 蜂窝）
3. 握手更快（常与 TLS 1.3 结合，少一次往返）
4. 解决 TCP 级队头阻塞：丢一包主要影响对应流，不堵死整连接

前提：客户端与 CDN/源站都支持；未支持时会回退到 HTTP/2 或 1.1。

---

## 对照总表


| | HTTP/1.1 | HTTP/2 | HTTP/3 |
| --- | --- | --- | --- |
| 传输 | TCP | TCP（多为 TLS） | QUIC / UDP |
| 多路复用 | 弱（多连接） | 强（单连接多流） | 强（且丢包更隔离） |
| 头部 | 明文重复 | HPACK 压缩 | QPACK 压缩 |
| 队头阻塞 | 应用层明显 | 应用层缓解，TCP 层仍有 | 更少 |
| DevTools Protocol | `http/1.1` | `h2` | `h3` |


---

## 开发者需要做什么

| 场景 | 建议 |
| --- | --- |
| 写业务代码 | 无需改 API；关注缓存、压缩、减少请求数 |
| 部署静态站 / CDN | 开启 HTTPS，让 CDN 开 HTTP/2 或 HTTP/3 |
| 性能分析 | DevTools Network → Protocol 列看 `h2` / `h3` |
| 本地开发 | `http://localhost` 常用 HTTP/1.1，属正常 |

前端优化在协议升级后仍重要：**减少关键资源、合理拆包、图片格式、缓存头**。HTTP/2 不等于可以无限碎文件。

---

## 如何在浏览器里确认

1. 打开任意生产站 → F12 → Network
2. 右键表头 → 勾选 **Protocol**
3. 刷新：常见 `h2`、`h3`、`http/1.1`

也可在 Security / 连接详情中查看 TLS 与协议版本。

## 常见误区

| 误区 | 正确理解 |
| --- | --- |
| 「HTTP/2 要改前端代码」 | 语义兼容；浏览器自动协商 |
| 「上了 h2 就不用管性能」 | 仍需控制资源体积与瀑布流 |
| 「HTTP/3 已全面取代」 | 覆盖在增，但会长期并存、自动回退 |
