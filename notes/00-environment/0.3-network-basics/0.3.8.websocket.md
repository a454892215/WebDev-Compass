# 0.3.8 WebSocket 与长连接的基本概念

> [← 返回 0.3 目录](./README.md) · [知识库地图](../../../knowledge-map/01-foundation-stages-0-2.md#03-网络基础web-开发者必备)

## 一句话理解

**WebSocket** 在一次 HTTP 握手升级后，建立 **全双工长连接**：客户端与服务器可随时互相推送数据，适合聊天、协作、行情、实时通知——不必反复发 HTTP 短请求轮询。

---

## HTTP 短请求的局限

| 方式 | 做法 | 问题 |
| --- | --- | --- |
| **短轮询** | 定时 `GET /messages` | 空转多、延迟受间隔限制 |
| **长轮询** | 服务器挂起直到有数据 | 仍是请求—响应模型，实现绕 |
| **SSE** | 服务器单向推流 | 客户端→服务器仍要另开 HTTP |

需要 **双方高频互推** 时，WebSocket 更合适。

---

## 连接如何建立

```
1. 浏览器发特殊 HTTP 请求（Upgrade: websocket）
2. 服务器返回 101 Switching Protocols
3. 之后该 TCP/TLS 连接上改走 WebSocket 帧
4. 双方可任意时刻 send / 收 message
```

URL：

- `ws://host/path` — 明文（仅开发或特殊环境）
- `wss://host/path` — TLS，**生产与 HTTPS 页默认用这个**

---

## 浏览器 API（精简）

```javascript
const ws = new WebSocket('wss://example.com/chat');

ws.addEventListener('open', () => {
  ws.send(JSON.stringify({ type: 'join', room: 'general' }));
});

ws.addEventListener('message', (ev) => {
  const data = JSON.parse(ev.data);
  console.log('recv', data);
});

ws.addEventListener('error', (err) => console.error(err));
ws.addEventListener('close', (ev) => {
  console.log('closed', ev.code, ev.reason);
  // 常在此做指数退避重连
});

// 主动关闭
// ws.close(1000, 'bye');
```

要点：

- `readyState`：`CONNECTING` / `OPEN` / `CLOSING` / `CLOSED`
- 载荷多为文本；也可传 `Blob` / `ArrayBuffer`
- **无内置自动重连与心跳**——生产需自己做或用库（如 reconnecting-websocket）

---

## 心跳与重连（工程常识）

```
客户端定时 ping（或发 {"type":"ping"}）
  → 服务器 pong
  → 超时未响应则认为断线 → close → 延迟重连
```

断线常见原因：网络切换、代理超时、进程休眠、负载均衡空闲断开。移动弱网尤甚。

## WebSocket vs HTTP（选型）


| | HTTP / REST | WebSocket |
| --- | --- | --- |
| 模型 | 请求—响应 | 长连接双向 |
| 适用 | CRUD、表单、文件上传 | 实时协作、推送、游戏同步 |
| 缓存 / CDN | 友好 | 一般不经传统静态 CDN |
| 鉴权 | Header / Cookie 自然 | 握手时带 Token；连接后也可首包鉴权 |
| 服务器成本 | 按请求 | 需维持连接状态 |


也可混合：**业务用 REST，通知用 WebSocket**。

还有 **SSE（Server-Sent Events）**：仅服务器→客户端，实现更简单，适合单向推送（行情简版、通知流）。

---

## 安全注意

1. 生产使用 **`wss://`**
2. 握手校验 **Origin**，防恶意外站滥用你的推送服务
3. 鉴权：Cookie（SameSite 注意）或 URL/`Sec-WebSocket-Protocol` 带短时 Token（避免长寿 Token 进日志）
4. 消息仍要做权限校验——连上不等于可订阅任意房间

## 📱 移动端对照

| Web | Android | Flutter | RN |
| --- | --- | --- | --- |
| `WebSocket` | OkHttp WebSocket | `web_socket_channel` | 同浏览器或原生库 |
| 重连 / 心跳 | 自行实现 | 自行或库 | 自行或库 |
| 后台存活 | 标签页可被挂起 | 需前台服务 / 系统推送 | 同左；冷启靠 Push |

App 后台常改用 **FCM / APNs** 推送，WebSocket 更适前台实时。

## 常见误区

| 误区 | 正确理解 |
| --- | --- |
| 「有实时需求就必须 WebSocket」 | 低频通知用轮询或 SSE 可能更简单 |
| 「连上一次就永远在」 | 必挂；必须设计心跳与重连 |
| 「WS 替代 REST」 | 多数资源操作仍适合 HTTP |
