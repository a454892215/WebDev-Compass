# 0.3.4 Cookie、Session、LocalStorage、SessionStorage 的区别与使用场景

> [← 返回 0.3 目录](./README.md) · [知识库地图](../../../knowledge-map/01-foundation-stages-0-2.md#03-网络基础web-开发者必备)

## 一句话理解

四者都在「记住状态」，但分工不同：**Cookie** 可随 HTTP 发给服务器；**Session** 是服务端会话；**localStorage / sessionStorage** 仅前端存数据，默认不随请求上传。

---

## 总览对比


| | Cookie | Session（服务端） | localStorage | sessionStorage |
| --- | --- | --- | --- | --- |
| 存储位置 | 浏览器（可 HttpOnly） | 服务器内存/Redis 等 | 浏览器 | 浏览器 |
| 容量 | ~4KB / 条 | 服务端决定 | ~5MB+ | ~5MB+ |
| 是否自动随请求 | ✅ 是（符合条件时） | 靠 Session ID（常在 Cookie 里） | ❌ 否 | ❌ 否 |
| 过期 | `Max-Age` / `Expires` | 服务端超时 | 永久（除非清） | 标签页关闭即清 |
| JS 可读 | 默认可；`HttpOnly` 则否 | 不可直接读服务端 Session | 可读 | 可读 |
| 同源范围 | Domain / Path / SameSite | 随 Cookie 策略 | 同源 | 同源 + **每标签页独立** |

## Cookie

浏览器按规则在后续请求的 `Cookie` 头中带上匹配的 Cookie。

### 关键属性

| 属性 | 作用 |
| --- | --- |
| `Name=Value` | 键值 |
| `Domain` / `Path` | 哪些主机与路径会带上 |
| `Max-Age` / `Expires` | 存活时间；都不设则为会话 Cookie（关浏览器可能清除） |
| `Secure` | 仅 HTTPS 发送 |
| `HttpOnly` | JS 读不到，降低 XSS 偷 Cookie 风险 |
| `SameSite=Strict\|Lax\|None` | 是否随跨站请求发送（防 CSRF；`None` 须配 `Secure`） |

服务端下发示例：

```http
Set-Cookie: sid=abc123; Path=/; HttpOnly; Secure; SameSite=Lax; Max-Age=86400
```

前端可读非 HttpOnly Cookie：`document.cookie`（字符串拼接，体验差；现代更推荐服务端会话或 Token）。

## Session（服务端会话）

典型流程：

```
1. 用户登录成功
2. 服务器创建 Session，存 userId 等，生成 sessionId
3. Set-Cookie: sid=<sessionId>
4. 之后请求自动带 Cookie → 服务器凭 sid 找回 Session
```

- **Session 本身在服务器**；浏览器只持有凭证（多为 Cookie 中的 ID）
- 也可把 Token 存 localStorage / 内存，用 `Authorization` 头传（无 Cookie 会话的另一种模式）

无状态 JWT 常见于前后端分离：服务器不存 Session，靠签名校验 Token——各有取舍（吊销、体积、密钥轮换）。

---

## localStorage

```javascript
localStorage.setItem('theme', 'dark');
localStorage.getItem('theme'); // "dark"
localStorage.removeItem('theme');
localStorage.clear();
```

- 同源共享，**关浏览器仍在**
- 仅存字符串；对象需 `JSON.stringify` / `parse`
- **不要存密码、长效 refresh token（尤其 XSS 风险高时）**
- 适合：主题、语言、非敏感偏好、草稿缓存

---

## sessionStorage

API 与 localStorage 相同，但：

- **每个标签页独立**
- 关闭标签页（或浏览器）后清除
- 适合：单页向导中间态、一次性表单草稿、标签页级 UI 状态

同窗口 `window.open` 同源页可继承 opener 的 sessionStorage 副本，行为需实测，勿强依赖。

## 选型指南

| 需求 | 推荐 |
| --- | --- |
| 登录态（传统 Web） | HttpOnly + Secure Cookie + 服务端 Session |
| 登录态（SPA + API） | HttpOnly Cookie（推荐）或内存 Token + 短效 Access Token |
| 主题 / 语言 | localStorage |
| 仅当前标签页的临时状态 | sessionStorage |
| 大量结构化离线数据 | IndexedDB（阶段 2.3） |
| 需随每个请求自动带 | Cookie（或自己加 Header） |

```
敏感凭证 ──优先──► HttpOnly Cookie（不给 JS）
偏好设置 ────────► localStorage
页内临时态 ──────► sessionStorage / 内存变量
```

---

## 安全要点（精简）

1. **XSS** 可读写 Storage 与非 HttpOnly Cookie → 敏感凭证优先 HttpOnly
2. **CSRF** 针对自动带 Cookie 的请求 → `SameSite` + CSRF Token
3. Storage **无自动过期** → 自己实现过期字段或登出清理
4. 隐私模式 / 用户清缓存会丢本地数据 → 不能当唯一信源

## 常见误区

| 误区 | 正确理解 |
| --- | --- |
| 「SessionStorage 就是服务端 Session」 | 完全不同；一个是浏览器标签页存储 |
| 「Cookie 过时了，全用 localStorage」 | Cookie 仍是会话与 CSRF 相关场景的核心机制 |
| 「存了 Token 到 localStorage 就安全」 | XSS 可直接偷走 |
